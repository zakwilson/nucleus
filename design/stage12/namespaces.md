# Modules and namespaces

Nucleus currently has two import forms, `import`, and `include`, both of which make all of a library's symbols available in the current file. This invites name collisions and makes all symbols public, limiting API design.

## Namespaces

The `ns` form sets the current namespace. A file or REPL session that does not set a namespace gets the default namespace 'user.

Calling import forms on symbols attempts to resolve a path to a file. 'foo.bar.baz could describe "foo/bar/baz.nuc", "foo/bar/baz.h", etc... similar to the current import resolution. Using a string argument resolves the filename by path relative to the library search path.

A namespace should be defined in a single file, aside from uses of `export`. More than one `ns` invocation, or calling `ns` with a namespace that already exists should be a warning at static compile time, but permitted silently in the REPL.

## Public and private

Global symbols are default public. `defn-`, `defvar-`, etc... create private symbols. Private always means the symbol is not importable from another namespace. For forms that emit a link symbol (`defn-`, `defvar-`) it additionally means internal LLVM linkage (the `static` analogue), so the symbol never enters the global link namespace and cannot collide at link time. This is where Nucleus finally gains internal linkage — storage-class specifiers were deferred in stage 8 (`design/stage888-deferred.md`). Compile-time-only forms carry no linkage dimension; for them private is pure visibility.

Every form that introduces a referenceable name gets a `-` variant: `defn-`, `defvar-`, `defconst-`, `defenum-`, `defstruct-`, `defunion-`, `defmacro-`, `defprotocol-`. Forms that name a *foreign* symbol (`extern`, `declare`) or assert a global fact with no name of their own (`extend`) have no private variant. `defcast` and `def-rmacro` register a global-effect fact rather than a name; a private variant would scope that fact to the namespace — coherent but niche, so deferred.

Types can be private. A private type that appears in (or is returned by) a public signature is allowed as an opaque export, but warns. The warning is suppressed under `import-use`, where the consumer has already opted out of the prefixing discipline.

### Import

This phase adds new import forms to control name collisions.

`import` takes an additional optional symbol parameter used as a prefix for symbols from the import. The sole parameter is used as the prefix if no optional prefix is specified. If library foo exports symbol bar, then (import 'foo 'baz) makes it available as `baz/bar` in the current namespace. This makes **all** current invocations invalid, forcing an intentional choice about which kind of import to use. Two imports may not share a prefix (error); the same library may be imported under different prefixes (aliasing). Import dedup is keyed on (file, prefix), not file alone.

`import-use` is the old import form, making all symbols available. Discouraged in production code. Good for REPL and libraries.

`import-only` imports a concrete list of symbols.

`unsafe-import-private` takes a namespace, prefix, and `&rest` private symbols. Obviously discouraged in normal situations. Alternately, this can go in the "unsafe" *library* and be `unsafe/import-private`.

The prelude remains `import-use`d by default. There are no specially blessed libraries or namespaces, but the compiler and the standard libraries may freely `import-use` their dependencies to avoid rewriting existing call sites — the prefixing discipline is for deliberate API design, not a mandate on internal code.

`include` gets removed. C headers come in through the same `import`/`import-use`/`import-only` forms with a string path: `(import "stdio.h" 'c)` exposes `c/printf`. Foreign symbols are never IR-mangled — they keep the link name fixed by the header regardless of the importing namespace's prefix, so `c/printf` still emits a call to `@printf`.

### Export

An `export` form re-exports an explicit list of symbols imported from another namespace. The symbol becomes visible under the host namespace's qualified name — a facade namespace `string` can `(export strview/find)` so consumers see `string/find` — reusing the original `ir-name`. No new code is emitted; it is an alias.

`set-ir-prefix` allows setting a prefix for IR mangling and consumption from C which is different from the Nucleus name for the namespace. It applies to the current namespace; a null/empty prefix means no prefix — symbols emit bare names, the escape hatch for hand-written C-ABI libraries.

## Compiler refactor

Most of the compiler is in a single file that's about ~14K lines long. That's a lot.

Once namespaces are stable, as much as practical should be moved out of the compiler core, given a clean public interface, and imported.

Files must be split off one at a time, and bootstrap/test verified after each one.

## Syntax

`/` and `_/` are already in use. A slash at the beginning or end of a symbol is part of that symbol's non-namespaced name. The **first** interior slash is the namespace divider; any later slashes belong to the name. So a bare `/` reads as the name `/` (the division operator — slash at both start and end), `_/` is untouched, and `foo/bar/baz` is the symbol `bar/baz` in namespace `foo`.

Slash is forbidden inside a namespace name, so `foo//bar` is unambiguously the symbol `/bar` in the namespace `foo`.

## Tooling

REPL tooling like apropos/locate/doc, as well as --emit-cheader get namespace support.

## Implementation

IR names are mangled, using either the namespace name or the `set-ir-prefix` call as a prefix. The scheme must be deterministic and produce C-legal identifiers (the qualified source name `foo/bar` is not one), so the divider lowers to something like `foo__bar`; existing method/instance mangling (`Name.tok.tok`, `Vector.i32`) composes after the namespace prefix. This is purely an ir-name concern — `Sym` already separates the source `name` from the emitted `ir-name` (`src/nucleusc.nuc:231`), so namespacing changes resolution, not the emitted-symbol machinery.

The default `user` namespace has an empty ir-prefix, so code that stays in `user` and `import-use`s its dependencies emits bare ir-names — byte-identical to today. The mangling only bites when a file declares a non-`user` namespace or sets a prefix. This makes the whole feature additive at the IR level: the compiler and libraries keep building unchanged until they deliberately opt into a namespace, so the bootstrap stays a fixed point through the stage. The breaking change is to the source surface (the import forms), not to emitted IR.

Foreign (C-imported) symbols are exempt: their ir-name is fixed by the header and is never re-mangled by the importing namespace's prefix.

Protocol and conformance facts stay **global**, keyed on fully-qualified type and protocol identity. Coherence is inherently global, and dispatch must resolve a conformance (e.g. `(extend coll/Vector coll/Seq)`) no matter which namespace triggers it.

## Robot — implementation status

**N1 (namespace core) — landed.** State, the `ns` form, and qualified resolution:

- `g-current-ns` (interned, default `"user"`) and a per-unit `g-ns-seen` flag, both initialized in `compiler-init` and saved/restored around each import in `emit-import` (an imported unit starts in `user`; the importer's namespace is restored afterward, so neither leaks across the boundary).
- `qualify-name` (next to `intern-str`) maps a source spelling to its interned storage key: split on the **first interior slash** (leading/trailing slash stays in the name, so bare `/`, `_/`, and leading-slash names are unqualified; `foo//bar` → namespace `foo`, name `/bar`); an unqualified spelling resolves against `g-current-ns`; the `user` namespace (and an explicit `user/` qualifier) stores **bare** keys.
- Storage keying is centralized in `scope-define`/`scope-lookup`: only the **global scope** (the unique null-parent scope) qualifies its key, so local bindings keep raw names and still shadow globals; `ir-name` is untouched (mangling is N4). Under `user`, `qualify-name` is the identity, so the whole bootstrap is byte-identical.
- `emit-ns` validates `(ns NAME)` (NAME a slashless symbol) and warns at compile time on a second `ns` in a unit (silent under the REPL flag `g-interactive`). A **leading** `ns` is consumed by `apply-leading-ns` at the top of `emit-toplevel-forms`, *before* the prescans, so global keys (and forward references) are qualified consistently from the first definition; `prepend-prelude-import` splices the auto-prelude *after* a leading `ns` so it stays the unit's first form. A non-leading `ns` is routed through the `'ns` dispatch case (and the REPL's `ns` branch). The reader is unchanged — `/` was already an ordinary symbol char, so `foo/bar` reads as one `NODE-SYM`.

Decision sharpened: the **storage-key** foreign-symbol exemption (decision 8) already bites in a hand-written `ns` + C-`include` file (the C-header dedup keys must stay bare so a re-`include` does not re-emit `declare @remove`); this is folded into N2/N4 as planned and does not affect the bootstrap (no compiler/library source declares a namespace yet).

**N2 (import forms) — landed.** The new import forms, additive over the unchanged legacy `import`/`include`:

- **`do-import`** is the single core driver (`prefix == null` → flatten; `prefix != null` → prefix-qualified). The old `emit-import` body became `emit-import-use`; `emit-import` is now a one-line flatten wrapper, so the legacy `import`/`include` keywords and the auto-prelude `(import prelude)` are **unchanged** — bootstrap stays byte-identical. New keywords: `import-use`, `import-only`, `import-prefixed`, `unsafe-import-private` (handlers `emit-import-*` to satisfy one-symbol-one-kind, since the keyword strings live in `g-special-form-set`). The bare `import` keyword is **not** yet flipped to prefix semantics — that is N7.
- **Prefix aliases**: after a prefixed import processes (or reuses) a file, `inject-import-aliases` walks the `[start, end)` slice of `g-globals` the file contributed and, per public symbol, `scope-define`s a `prefix/name` key carrying the **same** `type` and `ir-name` (`import-alias-one`). The alias is pure resolution — no IR, and a foreign C symbol keeps its bare `@name` (verified: `c/printf` → `call @printf`).
- **Dedup is two-level.** `g-imported` is the file-level loaded set (keyed on path, byte-identical), now recording the contributed slice in the pushed Node (`start` in `i`, `end` in `line`) so the **same file under a second prefix** re-injects aliases **without re-processing** (re-processing would re-emit its `defn`s and trip "duplicate method signature"). `g-import-aliased` keys on `(file, prefix)` (newline-joined) so a re-import under the same prefix is a silent no-op. `g-import-prefixes` binds prefix → file; a **second import claiming a bound prefix for a different file errors** (decision 5), checked *before* the file is processed so the diagnostic is the clean prefix conflict, not a downstream symbol clash.
- **C headers** flow through the string-path branch. Because a header may re-declare a symbol the prelude already surfaced (`<string.h>` already pulls in `printf` on glibc), the g-globals delta is **not** a reliable symbol list, so `emit-c-include` runs in **collect mode** (`g-cinclude-collecting`/`g-cinclude-collected`): it records every parsed function name — new or pre-existing — and `alias-cinclude-collected` aliases each via its existing bare `Sym`. The C-header path is exempt from the prefix-uniqueness rule (many headers under one `c/` is normal).
- **`import-only`** is flatten-with-an-intent-list for now: N2 has no per-import visibility gate (privacy is N3), so the listed symbols are guaranteed available but extras are not yet hidden. **`unsafe-import-private`** is exactly a prefixed import in N2 (no symbol is private until N3); its symbol list becomes the explicit skip-override once N3 marks `defn-`/`defvar-` private.

Decisions sharpened: (a) prefix uniqueness is a **pre-processing error** (clean prefix-conflict diagnostic), not a per-symbol alias-collision warning, which is now only a defensive guard. (b) Two libraries that define the **same bare symbol** still collide at the flatten level regardless of prefix — real isolation needs the importee to declare its own `ns` so N4 re-mangles its `ir-name`s; N2's prefix is a *resolution alias* layered over shared `user`-namespace definitions. The `example examples/import-prefix.nuc` exercises `c/printf`, prefixed `mathlib` under two prefixes, and `import-use` flatten; 103 tests pass, bootstrap byte-identical.
